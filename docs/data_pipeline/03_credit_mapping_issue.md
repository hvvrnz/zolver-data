### Issue: 전공선택 학점(major_elective) 값이 0으로 매핑되는 문제 발생 — 관측 지점 부족
초기 LLM 기반 구조화 과정에서는 전공선택 학점(major_elective)이 정상적으로 추출되었으나,
이후 단과대학명 구조화 및 JSON 출력 제약을 강화한 이후
모든 학과의 major_elective 값이 일괄적으로 0으로 출력되는 현상이 발생

#### 입력 조건:
- 동일한 PDF 원본
- 동일한 pdf_to_text() 결과
- 프롬프트에 다음과 같은 추가 제약이 포함됨:
  -  단과대학명 공백 제거
  - 학과명 공백 제거
  - 부연 설명 없이 JSON만 출력하도록 강제

#### 예시 출력 문제:
           {
          "department_name": "골프산업학과",
          "credits": {
            "general_education": 52,
            "major_required": 20,
            "major_elective": 0,
            "major_total": 72,
            "double_major": 0,
            "graduation_total": 132
          }
        }

      모든 학과의 'major_elective': 전공선택 학점이 0으로 고정됨



#### 원인:
현재 시점에서 정확한 원인은 단정할 수 없지만, 다음과 같은 가능성이 존재함.

1. LLM 출력 단계 문제 
  1-1. 제약 조건이 증가하면서 모델이 숫자 필드 중 일부(major_elective)를 생략하거나 기본값(0)으로 대체했을 가능성
  특히 “JSON만 출력” + “형식 엄격” 조건이 필드 보존보다 구조 준수에 우선되었을 가능성

2. 관측 지점 부족 문제
  2-1. 현재 파이프라인에서는

  PDF → text
  text → LLM input
  LLM output(JSON)  
  단계별로 필드 단위 값의 변화를 추적할 수 있는 로그가 없음

  2-2. 따라서 major_elective 값이
  LLM 입력에 있었는지
  LLM 출력에서 누락되었는지
  후처리 과정에서 0으로 대체되었는지를 눈으로 직접 구분해야 함
 
  -> 값 자체의 오류보다, 파이프라인의 규모가 커질 경우 어느 단계에서 값이 소실되었는지를 추적하기 어려워진다는 점이 더 큰 문제
  

#### 해결 방향:
1. 필드 단위 관측 로그 추가
   - major_elective를 주요 추적 대상 필드로 지정
   - 각 단계별 값 존재 여부 및 원시 값 기록
     (PDF 추출 결과 → LLM 입력 텍스트 → LLM 출력 JSON → 최종 후처리 결과)

2. Lineage 관점에서 데이터 흐름 정리 (예정)
   - major_elective 값이 생성·변환되는 흐름을 명시적으로 정의

3. 제약 조건 분리 실험 (예정)
   - 구조 제약 / 출력 제약 / 명명 규칙 제약을 분리
   - 어떤 제약이 정확도 하락에 영향을 주는지 단계적으로 검증
